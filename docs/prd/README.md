# Product Requirements Documents (PRDs)

Formal specifications for features and products.

## Structure

PRDs follow a consistent template:
1. Problem statement (user-centric, evidence-based)
2. Goals and success metrics
3. User stories and requirements
4. Constraints and assumptions
5. Implementation plan

## Naming Convention

- `0001-short-description.md` (sequential numbering)
- Include date in frontmatter
- Keep filename descriptive but concise

## Template

Use `docs/prd/templates/prd.md` for new PRDs.

## Workflow

1. Research phase (evidence gathering)
2. Draft PRD with problem statement
3. Review with stakeholders
4. Finalize and move to implementation (ADR/RFC)

## Key Principles

- Problem statements never reference tickets or filenames
- Every requirement traces to observed user evidence
- Success metrics are measurable
- Constraints are explicitly documented
